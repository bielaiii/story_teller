# Story Teller 产品功能路线图

状态：已确认方向，尚未实施
最后更新：2026-07-16
适用范围：Story Teller 写作工作台、Schema V3、本地编辑模式与静态只读模式

## 1. 目标与实施原则

Story Teller 已经具备人物、剧情、设定、碎片、关系图谱、时间线、篇章、回收站与智能提示等基础能力。下一阶段不继续堆叠彼此孤立的页面，而是围绕“构思、写作、检查、修改、成稿”建立完整闭环。

以下功能原则上建立在 Schema V3 的规范化实体、稳定 ID、外键关系、统一事务和增量 API 之上。除可独立完成的界面设计外，不应继续用扫描 Markdown、匹配名称或维护后续任务清单的方式实现新功能。

通用要求：

- 所有实体引用使用稳定 ID，显示名称只承担可读展示；
- 所有编辑原地完成，不整页刷新，不丢失编辑器、筛选、焦点、滚动与画布状态；
- 所有修改进入统一操作历史，并遵守撤销、冲突检测与七天保留规则；
- 功能必须同时打通界面、API、校验、SQLite 持久化、读回与错误状态；
- 静态部署允许查看已导出的结构化信息，但继续保持只读；
- 新增结构化字段尽量可选，不把创作工具变成强制填写的项目管理系统。

## 2. 实施顺序

建议顺序：

1. 完成 Schema V3 与增量 mutation 基础；
2. 全局引用与反向引用；
3. 双顺序视图与场景级信息；
4. 伏笔与回收管理；
5. 连贯性检查；
6. 全局搜索与命令面板；
7. 草稿与版本对比；
8. 成稿导出。

其中“全局引用与反向引用”是后续功能的数据基础；“伏笔与回收管理”是最直接提升创作价值的新增能力。

## 3. 全局引用与反向引用

### 3.1 目标

让人物、剧情、设定、地点、组织、物品、术语、篇章和时间线之间形成可以查询和跳转的稳定关系，而不是只在正文中出现同名文字。

### 3.2 用户能力

每个实体详情中显示：

- 被哪些剧情或正文使用；
- 出现在哪些篇章；
- 与哪些时间线节点和剧情线关联；
- 哪些人物关系、设定或伏笔依赖它；
- 引用位置附近的短预览；
- 点击后跳转到对应文章和正文位置。

编辑器通过 `@` 选择人物、通过 `/` 选择其他实体时，在插入可读文字的同时写入结构化引用。重命名只改变显示名称，不改变引用身份。

### 3.3 交互要求

- 反向引用放在现有详情卡片或详情页中，不新增常驻重型侧栏；
- 默认显示数量和少量最近引用，可展开查看全部；
- 点击引用保留来源页面的筛选和滚动状态，返回后回到原处；
- 已删除实体的引用不进入普通视图，但回收站预览可以说明其原有关联；
- 自然语言中恰好出现同名文字不自动视为精确引用。

## 4. 伏笔与回收管理

### 4.1 目标

把伏笔从普通备注升级为独立、可追踪的创作对象，帮助作者掌握“埋下、强化、误导、揭示、回收”的全过程。

### 4.2 数据与状态

每条伏笔至少包含：

- 稳定 ID 与标题；
- 简短说明；
- 首次埋下的位置；
- 中间强化或误导的位置；
- 计划回收位置；
- 实际回收位置；
- 关联人物、设定、剧情和时间线节点；
- 状态：未埋、已埋、推进中、已回收、废弃。

一次伏笔可以关联多个正文位置，正文位置引用必须能够跳转。

### 4.3 页面表现

- 在剧情或篇章详情中显示与当前内容有关的伏笔；
- 提供按状态、人物、篇章和剧情线筛选的总览；
- 未回收和临近计划回收的伏笔应清晰可见，但不使用打断写作的强制弹窗；
- 删除、恢复与撤销遵循统一实体和回收站机制。

## 5. 故事顺序与阅读顺序双视图

### 5.1 职责边界

- 故事发生顺序由时间线负责；
- 读者阅读顺序由篇章和剧情排序负责；
- 两者相互关联，但任何一方的排序修改都不能隐式重写另一方。

同一剧情可以较早发生、较晚揭露，从而自然支持倒叙、插叙、多视角回放和延迟揭秘。

### 5.2 用户能力

- 在时间线节点卡片中同时看到故事位置和阅读位置；
- 在剧情详情中查看该剧情位于哪条剧情线、何时发生、在哪一篇呈现；
- 并排或切换查看两种顺序；
- 从任一视图点击剧情，进入同一个剧情详情或正文；
- 拖动故事顺序只更新时间线排序，拖动阅读顺序只更新篇章内排序；
- 对尚未安排到某一顺序的剧情提供明确的“未安排”区域。

### 5.3 数据要求

故事排序键和阅读排序键分别保存。稳定 ID、用户看到的“第几篇”和两种排序位置不得混为同一个字段。

## 6. 连贯性检查

### 6.1 目标

在“配置检查”页面中增加面向小说内容的确定性诊断，优先发现可以由结构化数据证明的问题，不依赖生成式 AI 猜测。

### 6.2 首批规则

- 人物在同一故事时间出现在互相冲突的地点；
- 引用了已经删除、永久清理或不存在的实体；
- 图谱、时间线和结构化引用存在无效外键；
- 剧情时间、篇章顺序或时间线节点重复、缺失和明显矛盾；
- 剧情引用的人物、地点或物品在设定的登场时间之前出现；
- 伏笔已经埋下但长期没有推进或回收；
- 已标记回收的伏笔没有实际回收位置；
- 场景填写了互斥的角色定位或其他受约束属性；
- 同一人物档案的结构化事实之间发生明确冲突。

### 6.3 交互要求

- 每条问题说明原因、涉及对象和建议动作；
- 点击直接跳转到对应详情或正文位置；
- 区分错误、警告和提醒，不把创作选择误报成错误；
- 允许对特定提醒暂时忽略并写明原因；
- 检查只报告问题，不自动改写正文或悄悄修复数据；
- 数据库约束仍负责底层一致性，检查页面不能代替约束。

## 7. 全局搜索与命令面板

### 7.1 目标

提供一个贯穿所有页面的快速入口，统一查找人物、剧情、设定、碎片、伏笔、篇章、时间线和正文内容。

### 7.2 支持操作

- 搜索并跳转到实体或正文位置；
- 新建剧情、人物、设定、碎片或伏笔；
- 打开人物档案或剧情详情；
- 将当前剧情移动到某个篇章；
- 查看当前实体的反向引用；
- 在编辑器中插入结构化引用；
- 执行少量明确且可撤销的快捷操作。

### 7.3 检索要求

- 支持中文、全拼和拼音首字母；
- 结果按类型分组并显示简短上下文；
- 搜索结果不一次加载全部长正文；
- 编辑器中的 `@` 与 `/` 复用同一套索引和排序逻辑；
- 命令面板不得抢夺正在进行的中文输入法 composition；
- 选中结果后原地导航，不造成页面闪烁。

## 8. 草稿与版本对比

### 8.1 与操作撤销的区别

- 操作撤销用于处理近期误操作和数据修改；
- 创作版本用于保存作者主动选择的阶段性方案；
- 两者共享底层事务与 revision，但在界面和保留周期上明确分开。

### 8.2 用户能力

- 编辑期间自动保存草稿；
- 手动创建带名称和说明的检查点，例如“第二幕调整前”；
- 对比任意两个正文版本；
- 按整篇恢复，或只恢复用户选中的段落；
- 查看版本对应的时间、来源和关联操作；
- 恢复前预览差异，恢复动作本身仍可撤销。

### 8.3 状态保持

自动保存和创建版本都不能销毁编辑器实例，不得丢失光标、选区、输入法状态、编辑器撤销栈、同步滚动位置或沉浸模式。

## 9. 场景级信息

### 9.1 可选字段

剧情可以选择性记录：

- POV 人物；
- 故事时间或时间范围；
- 地点；
- 出场人物；
- 场景目标；
- 主要冲突；
- 场景结果；
- 情绪、节奏或张力；
- 关联伏笔。

### 9.2 设计原则

- 字段默认折叠，不压缩正文写作空间；
- 作者可以只填写有价值的字段；
- 人物、地点和伏笔使用结构化引用，不保存易失效的名称副本；
- 这些数据为时间线、连贯性检查、筛选和统计提供依据；
- 不把场景模板设为创建剧情的强制门槛。

## 10. 成稿导出

### 10.1 导出格式

- 完整 Markdown；
- Word；
- PDF；
- EPUB。

### 10.2 导出能力

- 严格按照当前阅读顺序合并正文；
- 自动生成篇章标题、章节标题和目录；
- 允许选择作品名、作者名、封面和基础排版；
- 可排除碎片、内部备注、伏笔说明、人物设定和其他创作辅助信息；
- 支持只导出选中的篇章；
- 导出前给出篇章缺失、空正文和顺序异常提示；
- 同一数据库 revision 和同一导出配置产生确定性结果。

导出文件是成稿产物，不是新的可写事实来源。修改导出文件不会反向覆盖 SQLite。

## 11. 后续可选方向

以下方向暂不进入近期实施范围，但 V3 设计不应阻塞它们：

- 基于本地选定上下文的写作辅助，但不得自动修改正文；
- 写作字数、节奏和阶段目标统计；
- 多作品之间复用设定模板；
- 面向外部编辑或读者的只读审阅包；
- 可选的批注与待处理建议。

不优先建设多人实时协作、在线账号体系或桌面应用壳。当前产品仍以本地单人创作、SQLite 完整备份和 GitHub 版本管理为核心。

## 12. 验收边界

任一功能只有同时满足以下条件才算完成：

- 界面中的每个入口都有实际可用的后端能力；
- 数据写入 Schema V3 的结构化表并可以重新读回；
- 正常操作不依赖 Markdown 扫描、文件名推断或后续修复清单；
- 保存、筛选、跳转和撤销不触发整页刷新或可见闪烁；
- 删除、恢复、冲突和失效引用都有明确行为；
- 关键业务规则有单元测试、HTTP round-trip 集成测试和必要的浏览器端到端测试；
- 静态只读模式不会显示不可用的写入控件；
- 文档、能力声明和实际运行服务版本保持一致。
